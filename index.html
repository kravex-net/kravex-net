How It Works

Three steps. One weekend.

Configuration-as-code. Validate before you commit. Resume if anything fails.

01 — CONFIGURE

Define your migration

Declare source, target, indices, and transformations in a YAML config. Version-controllable, reviewable, repeatable.

source:
  type: elasticsearch
  host: https://es-prod:9200
  indices: "logs-2024-*"

target:
  type: clickhouse
  host: https://ch-cluster:8443

transform:
  flatten_nested: true
  ilm_to_ttl: true
02 — VALIDATE

Dry-run everything

Pre-flight checks catch mapping errors, connectivity issues, and type incompatibilities before any data moves. No surprises.

$ kravex migrate --dry-run

Connecting to source... OK
Connecting to target... OK
Validating mappings... OK
Type compatibility... OK
Indices matched: 47
Estimated data: 3.2 TB
Estimated time: ~14 hours

✓ Ready to migrate
03 — MIGRATE

Run it. Go to bed.

Checkpoint/resume means if anything fails — network blip, node restart, disk full — you pick up exactly where you left off. Not from zero.

$ kravex migrate --config production.yaml

logs-2024-01 ████████████████ 100%
logs-2024-02 ████████████████ 100%
logs-2024-03 ████████████████ 62%
logs-2024-04 ████████████████ queued

Throughput: ~240 GB/hr | ETA: 6h 12m
